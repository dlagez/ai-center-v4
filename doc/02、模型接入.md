# 模型接入模块 PRD

## 1. 文档目的

定义 AI 中台的“模型接入模块”设计。

本阶段只做接入层抽象，不实现具体厂商调用细节。目标是先把模型接入模式、适配器接口、配置方式、注册机制设计清楚，支持后续用可配置、可插拔的方式接入不同模型来源。

本文档主要给代码实现使用，偏工程设计，不展开业务背景说明。

---

## 2. 模块目标

模型接入模块需要解决以下问题：

1. 屏蔽不同模型提供方的接入差异。
2. 支持多种接入方式，而不是绑定某一种 SDK。
3. 通过配置注册模型，而不是写死在代码里。
4. 支持后续按能力扩展，例如：
   - Chat Model
   - Embedding Model
   - Reranker Model
   - Image Model
   - Audio Model
5. 上层调用方不关心底层是 LiteLLM、LangChain、网关还是原生 SDK。
6. 后续可以平滑扩展鉴权、路由、超时、重试、限流、审计等能力。

---

## 3. 设计范围

### 3.1 本期范围

本期只定义以下内容：

- 模型接入方式分类
- 统一模型抽象
- 统一适配器接口
- 配置结构
- 适配器注册机制
- 模型注册机制
- 上层调用入口

### 3.2 本期不做

本期不实现以下内容：

- 具体厂商 API 对接
- 具体 SDK 调用逻辑
- 流式输出实现
- 函数调用/工具调用实现
- 图片/音频实际处理逻辑
- 负载均衡与智能路由
- 熔断、限流、降级
- 计费、配额、审计
- UI 管理界面

---

## 4. 接入设计原则

### 4.1 统一抽象

所有模型，无论来自什么来源，对上层都暴露统一接口。

### 4.2 接入方式与模型定义解耦

“怎么接”与“接什么模型”分开管理。

- Adapter 负责接入方式
- Model Profile 负责模型定义

### 4.3 配置驱动

模型、提供方、接入方式、能力声明都通过配置注册。

### 4.4 可插拔

新增一种接入方式时，只需要新增一个 Adapter，不影响现有调用方。

### 4.5 最小可行

先定义骨架，避免一开始设计过重。

---

## 5. 模型接入方式定义

模型接入模块支持以下接入方式。

## 5.1 LiteLLM SDK 接入

说明：

- 通过 LiteLLM 的 Python SDK 调用底层模型
- 适合统一接 OpenAI 兼容模型、云模型、多厂商模型
- 适合作为默认通用接入方式之一

标识建议：

- `litellm`

---

## 5.2 LangChain SDK 接入

说明：

- 通过 LangChain 的模型抽象接入
- 适合和 LangChain / LangGraph 生态集成
- 适合已经基于 LangChain 组织上层能力的场景

标识建议：

- `langchain`

---

## 5.3 OpenAI 兼容网关接入

说明：

- 对接统一网关地址
- 网关后面可以是 LiteLLM Proxy、new-api、自研网关、云代理等
- 平台侧只关心 OpenAI 兼容协议，不关心网关后面的厂商

标识建议：

- `openai_gateway`

---

## 5.4 自定义网关接入

说明：

- 对接非 OpenAI 协议的内部模型网关
- 适合企业内部统一模型服务
- 协议可能是 REST / RPC / HTTP SSE / WebSocket

标识建议：

- `custom_gateway`

---

## 5.5 原生 SDK 接入

说明：

- 直接接具体厂商 SDK
- 例如某些模型能力无法通过统一协议暴露时使用
- 适合作为特例扩展方式，不建议作为默认主路径

标识建议：

- `native_sdk`

---

## 5.6 本地运行时接入

说明：

- 对接本地推理服务或本地模型运行时
- 例如 vLLM、Ollama、LM Studio、自建推理服务
- 本质上也可以落到网关模式，但这里保留单独类型，便于语义区分

标识建议：

- `local_runtime`

---

## 6. 统一领域模型

---

## 6.1 Provider

表示模型来源或模型服务提供方。

示例：

- openai
- azure_openai
- anthropic
- qwen
- deepseek
- volcengine
- ollama
- vllm
- internal_gateway

建议字段：

- `provider_id`
- `provider_type`
- `name`
- `access_mode`
- `enabled`
- `default_config`

---

## 6.2 Access Mode

表示模型采用什么接入方式。

可选值：

- `litellm`
- `langchain`
- `openai_gateway`
- `custom_gateway`
- `native_sdk`
- `local_runtime`

---

## 6.3 Capability

表示模型能力类型。

可选值建议：

- `chat`
- `embedding`
- `rerank`
- `image_generation`
- `image_understanding`
- `speech_to_text`
- `text_to_speech`

一个模型可以声明多个能力，但首期建议一个模型实例只绑定一个主能力，简化实现。

---

## 6.4 Model Profile

表示平台里的一个“模型定义”。

它不是具体 SDK 对象，而是一份平台侧统一模型配置。

建议字段：

- `model_id`：平台内部唯一 ID
- `model_name`：显示名称
- `provider_id`：所属 Provider
- `access_mode`：接入方式
- `capability`：模型能力
- `upstream_model_name`：上游实际模型名
- `enabled`：是否启用
- `is_default`：是否默认模型
- `timeout`
- `max_retries`
- `meta`：扩展信息

说明：

- `model_id` 是平台使用的统一标识
- `upstream_model_name` 是实际请求上游时使用的模型名
- 上层调用尽量只依赖 `model_id`

---

## 6.5 Invocation Context

表示调用上下文。

首期先预留，不做复杂设计。建议字段：

- `tenant_id`
- `user_id`
- `request_id`
- `scene`
- `trace_id`
- `extra`

后续用于审计、路由、限额、链路追踪。

---

## 7. 核心模块设计

模型接入模块建议拆成以下几个核心部分。

---

## 7.1 Adapter 接口层

作用：

定义统一适配器协议，屏蔽不同接入方式差异。

核心职责：

- 根据模型配置初始化客户端
- 接收统一请求对象
- 返回统一响应对象
- 屏蔽底层 SDK / HTTP 调用差异

建议抽象：

- `BaseModelAdapter`
- `ChatModelAdapter`
- `EmbeddingModelAdapter`
- `RerankModelAdapter`

说明：

- 首期可以只做一个 `BaseModelAdapter`
- 后续按能力拆分子接口

---

## 7.2 Adapter Registry

作用：

管理“接入方式”到“适配器实现类”的映射关系。

示例：

- `litellm` -> `LiteLLMAdapter`
- `langchain` -> `LangChainAdapter`
- `openai_gateway` -> `OpenAIGatewayAdapter`

职责：

- 注册适配器
- 根据 `access_mode` 查找适配器
- 支持后续插件化扩展

---

## 7.3 Provider Registry

作用：

管理 Provider 配置。

职责：

- 加载 provider 配置
- 查询 provider
- 校验 provider 是否启用
- 向 adapter 提供 provider 级默认配置

---

## 7.4 Model Registry

作用：

管理平台模型定义。

职责：

- 加载模型配置
- 按 `model_id` 查询模型
- 获取默认模型
- 按 capability 过滤模型
- 校验模型与 provider 的关联关系

---

## 7.5 Model Service

作用：

作为上层统一入口。

上层应用只和 `ModelService` 交互，不直接依赖具体 Adapter。

职责：

1. 接收调用请求
2. 根据 `model_id` 找到 `Model Profile`
3. 根据 `access_mode` 找到对应 Adapter
4. 组装调用参数
5. 执行调用
6. 返回统一结果

这是模型接入模块的核心门面层。

以下是逻辑接口，不要求现在就写完整实现。

## 9. 统一配置设计

建议采用声明式配置，支持 YAML / JSON。

首期推荐 YAML。

---

## 9.1 Provider 配置示例

```yaml
providers:
  - provider_id: openai_default
    provider_type: openai
    name: OpenAI 官方
    access_mode: litellm
    enabled: true
    default_config:
      api_key_env: OPENAI_API_KEY
      base_url: https://api.openai.com/v1

  - provider_id: internal_gateway
    provider_type: gateway
    name: 内部模型网关
    access_mode: openai_gateway
    enabled: true
    default_config:
      api_key_env: INTERNAL_GATEWAY_API_KEY
      base_url: http://model-gateway.xxx/v1
